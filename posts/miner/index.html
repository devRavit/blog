AI 에이전트로, AI 에이전트를 만들다 — miner 개발 라이프사이클 회고

국내(KRX)+미국 주식 자율매매 에이전트 miner를 3주에 만들며 있었던 일. 이 글의 주제는 "무엇을 만들었나"보다 "어떻게 만들었나" — 제 개발 방식입니다. 이야기의 두 심장은 LLM AI 엔지니어링(LLM·Koog·vector DB로 에이전트를 만든 기술 — 셋 다 처음이었습니다)과 Loop 엔지니어링(그 에이전트를 만든 개발 방식)입니다.

한눈에 (TL;DR)

무엇 — 혼자 3주에 LLM 자율매매 에이전트를 만든 이야기. 기술적으로 가장 새로웠던 두 축:

  • LLM AI 엔지니어링 = 에이전트를 (처음) 만들기. LLM 통합(Koog)·structured output·메멘토 페르소나·vector DB 회상(RAG) — LLM도 Koog도 vector DB도 다 처음이었다. 한마디로 "에이전트를 프로그래밍한다."
  • Loop 엔지니어링 = 그 에이전트를 만든 개발 방식. 단발 "guess once"를 "try → check → fix → repeat" 로. 검증(테스트·리뷰)을 루프의 한 단계로 내장하고, 되돌리기 어려운 것만 사람이 잡는다.
  ┌─▶ 이슈(목표) → 코드+테스트(행동) → validate·CI(관측) → 7관점 리뷰(검증) ─┐
  │                     ▲ 게이트 실패 시 되돌아 반복                           │
  └── 머지·close(종료) ← 흡수·교정(복구) ←───────────────────────────────────┘

실측(3주) — /goal 킥오프 22번 → assistant 턴 82,959번. 토큰 42B, 그중 99%가 캐시 read(→ 비용은 "출력"이 아니라 "맥락 × 턴 수"가 결정). 자율성 31:1(1시간 이상 무개입 122회, 최대 12.7시간).

혼자서 3주에 이 밀도가 나온 건, 제가 코드를 한 줄씩 다 쳤기 때문이 아닙니다. AI 에이전트를 개발자로 세우고, 그 에이전트가 도는 개발 루프를 엔지니어링했기 때문입니다. 이 글은 그 방법론의 기록입니다 — 잘된 부분뿐 아니라, 질책받고 고쳐간 과정까지 솔직하게.

1. 무엇을 만들었나 (배경 한 컷)

먼저 대상만 짧게. miner는 LLM이 스스로 판단하는 자동 주식매매 에이전트입니다. 흔한 자동매매(사람이 조건식을 짜고 기계가 실행)와 달리, 매 사이클 LLM이 뉴스·시세를 종합해 사람 트레이더처럼 추론해서 사고 팝니다. 무엇을 매매할지 + 다음에 언제 다시 생각할지까지 스스로 정하는 self-scheduling 에이전트죠.

안전 모델은 3층입니다 — 도구·관측(결정론) → 판단·집행(AI) → 리스크 가드(결정론, 절대 AI에 안 맡김). AI는 판단만, 집행은 반드시 결정론 가드를 통과.

이 에이전트를 어떤 기술로 만들었나는 §4(LLM·Koog·vector DB), 어떤 개발 방식으로 만들었나는 §5(루프)에서 다룹니다. 트레이딩 시스템 전체 골격과 마일스톤은 §7입니다.

2. 출발점 — 비전을 코드가 아니라 "문장"으로 박다

가장 먼저 한 일은 코드가 아니라 PRD(요구사항 문서) 였고, 여기서 스스로에게 규칙을 걸었습니다: PRD에는 요구사항(WHAT)만, 개발 결정(HOW)은 전부 ADR에.

PRD에는 제가 던진 비전을 원문 그대로 박았습니다:

"말그대로 주식 트레이더의 하루일과를 ai로 바꾼다 이게 골격이야" "냉혈한 안경잡이 주식트레이더 컨셉 가자" "근데 이건 순수 개발자인 내 머리에서 존재하는 환각의 주식 트레이더의 일과야."

왜 원문을 고집했을까요. AI와 협업해 문서를 쓰다 보면 요구사항을 슬쩍 깎거나, 학술 용어를 발명하거나, 아직 안 정한 걸 다 정해진 것처럼 써버리기 쉽기 때문입니다. 그래서 작성 규약을 못박았습니다 — 발언은 원문 인용, 숫자 발명 0, 안 한 말 발명 0, 미결정은 "결정 대기"로 명시.

이게 개발 방식의 첫 원칙입니다: AI에게 맥락을 줄 때, 원본을 왜곡 없이 보존한다. 비전은 안 흔들리는 북극성이고, 67개의 ADR이 "그래서 어떻게 만들 것인가"의 격전지가 됩니다.

북극성 — 어떤 마인드로 시작했나

시작할 때의 마음가짐은 "규칙 몇 개로 돌아가는 매매 봇을 만든다"가 아니었습니다. 트레이더의 하루 일과를 통째로 AI로 옮긴다 — 뉴스를 읽고, 종목을 고르고, 사고팔고, 장 마감에 회고하고, 다음에 언제 다시 볼지까지 스스로 정하는, 사람처럼 판단하는 에이전트를 만든다는 쪽이었습니다. 그래서 처음부터 "코드를 짠다"보다 "에이전트를 프로그래밍한다" 는 프레임으로 접근했습니다.

북극성은 더 멀리 있었습니다. 이 에이전트가 신입처럼 수습을 거쳐 점점 자립하는 단계를 그렸습니다:

Phase 상태 전환 게이트
0 수습 PAPER(가상돈) · 가상 체결만 KPI net-positive 입증
1 정직원 LIVE 소액 · 한도·승인 매매 net-positive
2 자립준비 LIVE + 트레저리 —
3 자가운영 수익으로 자기 인프라를 자동 결제 —

마지막 단계 — 에이전트가 자기 수익으로 자기 서버비를 내는 자가운영 — 이 이 프로젝트의 북극성이었습니다. 3주로 거기까지 갈 순 없지만, 안전 3층·KPI·회고 학습 같은 설계 결정이 전부 그 방향을 향하도록 잡았습니다.

결정은 ADR로 — 누가 쓰고, 어떻게 나오고, 무엇이 오가나

솔로 메인테이너 + AI 협업의 최대 약점은 세션이 끝나면 "왜 그렇게 정했는가"가 증발한다는 것입니다. 그래서 중요한 결정은 전부 ADR(Architecture Decision Record, MADR 포맷)로 docs/decisions/에 영구 기록했습니다. 트리거는 아키텍처 패턴·인프라 선택·외부 의존 도입·정책 변경.

누가 쓰나. 결정자는 언제나 저(단독 메인테이너)입니다. 하지만 ADR 초안은 개발 에이전트가 씁니다 — 제 PRD 원문과 코드 정독, 웹 실증, 서브에이전트 다관점 논의를 종합해서. 저는 리뷰하고 정정합니다. 어떤 ADR(0046 미장 정보/투자 분리)에는 제가 5번 반복 정정한 흔적이 그대로 남아 있고("미장 정보 는 깎지 마라, 미장 투자 만 막아라"), 그 정정 횟수 자체가 "결정의 무게"로 기록돼 이후 작업에서 최우선 보존됩니다.

어떻게 나오나. ADR은 추측으로 쓰지 않습니다. LLM 프로바이더를 정한 ADR(0025)이 좋은 예입니다 — 제가 "무료 티어로는 1시간도 못 버틸 것 같다, 뉴스 수집에서 터질 듯"이라고 직감을 던지면, 에이전트가 코드를 정독해 실제 LLM 호출 빈도를 정량화(테마 스캔 일 1회 / 협상 사이클 최악 12회…)하고 당시 기준 rate limit을 웹으로 대조해 판정합니다. 결론(무료론 무리)은 제 직감이 맞았지만 병목 대상은 빗나갔습니다 — 뉴스가 아니라 협상 RPM burst였죠. 이런 걸 실증으로 교정합니다.

무엇이 오가나. 포맷은 다섯 칸 — 맥락(요구·동기) / 결정(채택안) / 결과(얻는 것·잃는 것) / 대안(왜 그건 안 했나) / 관련. 그리고 상태가 흐릅니다: 제안 → 채택 / 부분 수락 / 보류 / 대체됨 / 폐기. "제안"은 써두되 항목별 채택은 후속 PR·이슈에서 확정한다는 뜻이고, "부분 수락"은 선행 조건이 필요한 항목만 남겨둔 것입니다(0046은 집행 시점 가드를 먼저 넣고, 카탈로그 가드는 주문 어댑터 완성 뒤로).

진실이 아니라 역사다. 이게 핵심 규율입니다. ADR은 "그 시점의 결정 기록"이지 현재 진실이 아닙니다. 현재 진실은 가이드북(domain·architecture·conventions). ADR끼리 충돌하면 번호 높은 쪽이 최신이고, 채택된 ADR은 다시 열지 않고 새 ADR로 supersede합니다. 번호는 영구 식별자라 삭제돼도 재사용하지 않고 결번(tombstone)으로 남깁니다. 그래서 매매 회로를 재설계할 때 낡은 ADR 10개를 지우면서도 번호는 비워두고 "이 개념은 어디로 대체됐다"는 표지를 남겼습니다 — 틀린 결정을 지우되, 지웠다는 사실은 남긴다.

3. 개발 라이프사이클 — 한 번의 반복(iteration)의 해부

제 개발 방식은 하나의 반복 루프입니다. 이슈 하나가 도는 한 바퀴는 이렇습니다:

이슈(의도) → branch → 개발+테스트 → validate → PR(10섹션) → CI → 7관점 리뷰 → 흡수/후속분리 → rebase merge → 자동 close

각 화살표마다 통과하지 못하면 다음으로 못 가는 문(게이트)이 달려 있습니다. 주요 관문만 짚으면:

  • 이슈 = 의도지 명세가 아니다 — yml 키·기본값·클래스명을 박지 않고 "왜/무엇/대안"만. 착수 시 이슈 본문과 연관 ADR을 직접 정독(요약 위임 안 함).
  • validate — ./gradlew validate가 lint+build+kover 100%+version+CHANGELOG를 강제. pre-push git hook에도 걸려서, 통과 못 하면 push 자체가 안 됨.
  • PR 10섹션 + main ruleset — PR 본문 10개 섹션(Why/의사결정/패턴/…)을 CI가 검사. main엔 ruleset(force-push·non-rebase 차단, CI 미통과 머지 차단, bypass_actors: [] — 나 자신도 우회 불가).
  • 7관점 병렬 리뷰 — 머지 전 7개 관점(시니어·PM·도메인 마스터·트레이더·인프라·보안·문서)을 동시에 띄워 GitHub inline comment로 등록.
  • 2단 배포 — main 머지 → miner Deploy(ECR) → deployment-hub(SSH pull/restart) → health 폴링 → /info 버전 검증. "빌드 성공 ≠ 서비스 교체 완료."

게이트는 "발견"되는 순간 만든다

이 게이트들은 처음부터 다 있었던 게 아닙니다. 한번은 PR 템플릿을 리팩터링하다 PR 본문 검사 로직이 함께 삭제됐고, 그 뒤 PR 하나가 10섹션이 아니라 2섹션만으로 CI를 통과해버렸습니다. "CI에 다 막아둔 것 같은데 어떻게 통과된 거야?" — 이 순간의 교훈이 제 방법론의 척추가 됐습니다:

자율 규약은 휘발된다. "앞으로 이렇게 하자"는 다짐은 반드시 잊힌다. 그러니 규칙을 발견하는 즉시 CI/hook 게이트에 박는다.

"파일 150줄 미만" 같은 자율 룰도 임계값(150)조차 위반으로 보고 Konsist spec으로 강제했습니다. 문서에만 적힌 규칙은 안 지켜지고, 게이트에 박힌 규칙만 살아남습니다. 문서는 그 게이트의 단일 진리원 역할만 합니다.

(GitHub ruleset 자체에도 함정이 많았습니다 — required_status_checks는 머지뿐 아니라 push 시점에도 적용돼서, 워크플로 트리거를 짝 안 맞추면 push가 영원히 차단됩니다. 이런 걸 하나씩 부딪혀가며 게이트를 다졌습니다.)

같은 사고가 아키텍처 선택에도 작동했습니다. 코드 패턴을 헥사고날(포트 & 어댑터) 로 잡은 이유 중 하나는, 그게 외부에서든 내부에서든 워낙 유명하고 구체적으로 정형화된 패턴이라 레이어 격리를 Konsist로 기계적으로 강제하기 쉽기 때문입니다. "코어는 외부 의존이 0이어야 한다", "어댑터는 포트를 통해서만 코어에 닿는다" 같은 규칙을 사람이 매번 지키는 대신 정적 검사로 못박을 수 있으면, 그건 곧 AI에게 줄 수 있는 게이트가 됩니다. 즉 아키텍처를 '지켜야 할 자율 규약'이 아니라 '어길 수 없는 강제 게이트'로 만들 수 있느냐가 선택 기준의 하나였습니다 — 유명하고 정형화된 패턴일수록 그 강제 표면이 넓습니다.

이 흐름 위에서 실제로 개발을 굴리는 두 엔지니어링이 §4·§5입니다.

4. LLM AI 엔지니어링 — 에이전트를 (처음) 만들다

이 프로젝트에서 기술적으로 가장 새로웠던 축입니다. LLM을 코드로 다루는 것도, Koog라는 에이전트 프레임워크도, vector DB도 전부 처음이었습니다. "매 사이클 스스로 판단하는 트레이더 에이전트"를 만든다는 건, 이 셋을 엮어 하나의 판단하는 뇌를 짓는 일이었습니다.

LLM을 코드로 — Koog와 듀얼 프로바이더

에이전트 프레임워크는 Koog(JVM 네이티브)를 골랐습니다. 스택이 Kotlin/Spring이라 JVM에서 그대로 도는 게 컸습니다. 처음엔 LLM 응답에서 JSON을 손으로 긁었는데(extractJson), 모델이 형식을 조금만 어겨도 깨졌습니다. Koog native structured output으로 옮겨 출력을 타입으로 강제하고서야 안정됐습니다. 프로바이더는 둘을 병행합니다 — 자주 부르는 관측·중간 추론은 Gemini 2.5 Flash(싸게), 최종 판단은 GPT. 여기서 처음 만난 함정이, 프로바이더마다 structured output의 strict 스키마 규칙이 달라서 프로바이더별 스키마 생성기를 따로 둬야 했다는 것입니다.

에이전트에게 "판단"을 시키는 법 — tool-calling 루프

핵심 설계는 에이전트를 고정 절차의 부품이 아니라 도구를 쥔 판단자로 두는 것이었습니다. ToolCallingLoop이 LLM의 도구 호출(관측·회상·거시·뉴스, 그리고 매매·관망·재예약)을 받아 중계하고, 에이전트가 "지금 뭘 볼지, 뭘 할지"를 스스로 고릅니다. 다만 LLM에 전부 맡기지는 않았습니다 — RuleEngine으로 "무엇을(WHAT)은 LLM, 언제·얼마나(WHEN·HOW MUCH)는 규칙"으로 역할을 갈랐습니다. 판단은 열되, 위험한 결정은 결정론이 잡게.

에이전트의 인격 — 메멘토 페르소나와 프롬프트 외부화

처음 페르소나는 "당신은 기획팀입니다" 한 줄이었는데, 상황도 동기도 빈약했습니다. 이걸 메멘토 서사로 다시 세웠습니다 — 매 사이클 기억을 잃고 깨어나, 스스로 남긴 단서(관측 스냅샷·저널·회상 메모)로 상황을 다시 조립하는 트레이더. 이 서사가 stateless 사이클 + 영속 메모리라는 실제 구조와 정확히 맞고, "기록에 없는 건 환각하지 말고 단서로만 취급하라"는 anti-hallucination을 규칙이 아니라 캐릭터의 성격으로 내면화시켰습니다. 프롬프트는 코드 상수가 아니라 resources/prompts/*.txt로 외부화하고, prompt_versions 컬렉션에 버전을 둬 배포 없이 갈아끼우게 했습니다.

vector DB, 첫 경험 — 기억하는 에이전트 (RAG)

가장 낯설었던 건 vector DB였습니다. "에이전트가 과거 판단을 회상한다"를 어떻게 구현하나 — 답은 MongoDB Atlas Vector Search였습니다. 임베딩이 뭔지부터 익혔습니다: 텍스트를 벡터로 바꾸고(768차원), 의미가 가까운 것을 $vectorSearch로 찾는 것이죠. 여기서 초심자다운 함정을 몇 개 밟았습니다. 처음엔 임베딩 어댑터가 껍데기(NullEmbeddingAdapter, 빈 벡터)라 검색이 늘 무결과였고, 나중에 실 어댑터로 바꿨는데 그 토글이 꺼지면 부팅은 정상인데 회상만 조용히 죽는 함정이 또 있었습니다(§9). 회상을 붙이고 나서는 두 가지 병을 막아야 했습니다 — 사후확신(결과를 알고 판단을 미화)은 결정 시점의 관측 스냅샷을 그때 임베드해 차단하고, 결과편향(어쩌다 맞은 판단 승격)은 장 마감(EOD) 회고에서 반복된 패턴만 승격하도록 게이트를 걸었습니다.

에이전트의 행동을 어떻게 설계했나 — 증권사 기능 역산

에이전트가 "트레이더처럼" 행동하게 하려면 도메인 감각이 필요한데, LLM 일반지식만으론 빈틈이 컸습니다. 그래서 새 개념(watchCondition·주문 유형·리스크 관리)을 설계할 때마다 실제 증권사 앱/HTS 기능을 웹으로 대조했습니다. 논리는 이렇습니다 — 증권사가 그 기능을 만든 건 실수요가 있어서이고, 그 수요가 트레이더의 실제 요구이며, 그 요구의 뿌리가 트레이더의 관점이니, 거기서 "AI 트레이더가 어떤 태도로 행동해야 하는지"를 역산할 수 있다는 것. 증권사 기능은 검증된 행동 패턴의 화석이라, 추측 대신 그 위에 에이전트의 행동 모델을 세웠습니다.

세 가지(LLM·Koog·vector DB)가 다 처음이었다는 게 이 축을 가장 새롭고 어렵게 만들었고, 동시에 가장 많이 배운 곳이었습니다.

5. Loop 엔지니어링 — 그 에이전트를 만든 개발 방식

두 번째 축은 그 에이전트를 만들어낸 개발 방식, 즉 Loop 엔지니어링입니다. 이건 LLM 앱 개발의 한 분과이기도 합니다. AI 애플리케이션 개발은 이렇게 진화해 왔습니다:

프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → harness 엔지니어링 → Loop 엔지니어링
   (좋은 한 번의 응답)     (무엇을 보여줄까)     (도구·권한 인프라)    (반복 사이클을 설계)

Loop 엔지니어링은 단발 응답을 "행동→관측→검증→복구→반복→종료"의 목표 지향 사이클로 바꾸는 것입니다. 잘 만든 루프는 "guess once"(한 번에 맞히기)를 "try, check, fix, repeat"(해보고, 확인하고, 고치고, 반복) 로 바꿉니다. 검증 가능한 결과가 있는 작업에서 이건 긴 프롬프트 한 방을 압도합니다.

그리고 핵심은 이겁니다 — 루프는 마법이 아니라 엔지니어링 산출물입니다. 그래서 저는 이 개발 루프를 하나의 재사용 가능한 명령어(/goal)로 코드화했습니다. 매 개발 사이클을 그 명령어 하나로 킥오프합니다.

제 개발 라이프사이클(§3)이 정확히 이 루프였습니다. ReAct 패턴(Reason→Act→Observe→Repeat)에 매핑하면:

Loop 엔지니어링 요소 제 개발 루프에서의 구현
명확한 목표·종료조건 이슈(의도) + "머지·close" 라는 테스트 가능한 종료조건
도구 접근 git·gradle·gh·테스트 러너·서브 에이전트
행동(Act) 메인 에이전트가 코드+테스트 작성
관측(Observe) validate·CI 결과, 커버리지 리포트
검증(Verify) 7관점 리뷰 + kover 100% (매 반복의 verify 단계)
복구(Recover) 리뷰 발견 흡수·교정, 실패를 신호로 되먹임
종료(Terminate) CLEAN 확인 후 rebase merge → 자동 close

개발 에이전트를 모는 규율

루프가 잘 돌려면 그 안에서 코드를 쓰는 개발 에이전트를 잘 세팅해야 합니다. 프롬프트(/goal)는 얇게 뒀습니다 — "규칙을 담지 않고, 문서가 정의한 게이트를 집행하는 오케스트레이션일 뿐. 숫자·관점·기준은 해당 문서를 읽어 따른다." 프롬프트에 박은 숫자는 문서와 어긋나는 순간 거짓말이 되니까요. 대신 맥락은 docs/를 단일 진리원으로 두고(modules.md는 코드와 1:1 미러, ADR·컨벤션), 에이전트가 매 반복 필요한 문서를 읽어 재조립하게 했습니다. 코드는 메인 에이전트가 직접 씁니다 — 한번 큰 기능을 서브 에이전트에 통째로 위임했다가 "너가 직접 구현해, 앞으로 에이전트 써서 개발은 하지마" 라는 답을 들었습니다. 서브 에이전트는 조사·리뷰·관점 소환 담당이고, 구현을 위임하면 주인의식·정독·맥락이 옅어지니까요. 그리고 쓰기 전엔 파급을 전수 grep하고 커밋 전엔 diff를 자가 리뷰해서, validate·리뷰를 "발견 수단"이 아니라 최후 방어선으로만 씁니다 — "리뷰 지적당하기 전에 한두 번 더 고민."

검증을 루프에 내장한다

핵심은 검증이 사람의 눈이 아니라 루프의 한 단계라는 점입니다. kover 100%·CI·7관점 리뷰가 매 반복의 verify를 담당합니다. 초기 구현이 틀리면 테스트가 잡게 두고 교정합니다(점진 자가검증) — 지레짐작하거나 템플릿을 복붙하지 않고, 단계마다 추론을 쌓으며 루프가 스스로 오류를 드러내게 합니다.

검증의 품질 — 커버리지의 "원리"부터 "무엇을 검증할지"까지

테스트가 verify 단계를 떠맡으니, 테스트 품질이 곧 루프의 신뢰도입니다. 여기서 100% 커버리지는 목표가 아니라 최소 바닥일 뿐이었고, 두 가지를 따로 파고들어야 했습니다.

① 커버리지가 어떻게 측정되는지 알아야 100%가 의미 있다. kover(JaCoCo)는 line과 branch를 각각 셉니다. 이 측정 원리를 모르면 99.9x%에서 영영 막힙니다 — 람다 경유로 호출된 함수는 미커버로 잡히고, sealed when은 컴파일러가 만든 도달불가 else 때문에 branch가 안 채워지고, Nothing 반환 헬퍼를 분기에서 호출하면 팬텀 fall-through 엣지가 남습니다(직접 throw로 바꿔야 사라짐). 측정 메커니즘을 이해해야 커버리지를 "게임"하지 않고 진짜로 채웁니다.

② 커버리지 100%여도 "무엇을 검증했나"는 별개다. 라인이 실행됐다 ≠ 로직이 맞다. assertion이 동어반복이면 커버리지는 초록불인데 실제론 아무것도 못박지 않은 것입니다. 그래서 커버리지와 별도로 "무엇을 검증할지"를 도출합니다 — 임계값의 경계, 변이 검증(값을 일부러 바꿨을 때 테스트가 깨지나 — 안 깨지면 그 테스트는 가짜다), 이 슬라이스가 진짜 못박아야 할 불변식. 코드를 직접 정독해 도출하지, 템플릿을 복붙하거나 이전 spec을 캐싱하지 않습니다.

두 축을 다 통과해야 verify가 믿을 만해집니다 — 커버리지는 "빠짐없이 봤나", 검증 도출은 "제대로 봤나". (그리고 유닛이 잡는 건 여기까지 — 실제 동작의 최종 확인은 e2e입니다. §9)

반복 간 컨텍스트 관리 — 폐기된 전제의 잔재를 지운다

루프가 길어지면 과거의 실패한 접근이 되살아나는 게 제일 위험합니다. 설계가 진화하면(전종목 발굴 폐기, 결정론 DSL → LLM 시맨틱 전환 등) 이전에 쓴 이슈들이 죽은 전제를 담은 채 남고, 그대로 착수하면 역행 구현이 나옵니다. 그래서 이슈 하나를 close한 직후, 다음 이슈 착수 전에 열린 이슈 전체를 최신 결정과 크로스체크합니다 — 죽은 전제는 닫고, 부분 유효는 본문을 갱신. 점검 대상은 이슈만이 아니라 코드 주석·KDoc·제 메모리까지 포함합니다. 문서 단일 진리원이 여기서 "compact working memory" 역할을 합니다.

루프를 병렬로 — 워크트리 N개 + wip 라벨 락

한 루프로는 못 따라가는 순간이 왔습니다. 7관점 리뷰가 PR마다 새 발견을 뱉고, 개발하다 보면 파생 이슈가 계속 생겨서 — 이슈가 쌓이는 속도가 처리 속도보다 빨랐습니다. 그래서 루프 자체를 병렬로 돌렸습니다: Claude 세션 여러 개를 동시에, 각각 별도 git worktree(~/repository/miner, miner-second, miner-592, miner-736…)에서.

문제는 조율입니다. 세션끼리 직접 통신할 방법이 없습니다 — 원격 GitHub이 유일한 공유 진실이죠. 그래서 GitHub 이슈의 wip 라벨을 세션 간 락으로 썼습니다:

  • 착수 전: wip 라벨·관련 브랜치·PR이 있는지 확인. 하나라도 있으면 다른 세션이 잡은 것 → 건너뛰고 다음 이슈. (오래 방치된 남의 PR도 함부로 rebase·머지하지 않습니다 — 다른 세션 작업물 오염.)
  • 착수 즉시: 브랜치를 만들기 전에 wip 부착. 락 없이 시작하면 그 짧은 창에서 다른 세션이 같은 이슈를 잡습니다.
  • 완료 시: 머지로 이슈가 자동 close돼도 라벨은 자동 해제가 안 됩니다 → 마지막 스텝으로 명시적 제거.

그리고 병렬은 자원을 놓고 부딪힙니다. 워크트리들이 gradle 데몬을 공유해서 한쪽 --stop이 옆 빌드를 죽이고, 임베디드 Mongo가 고정 포트를 물어 테스트가 동시에 터졌습니다(이 둘의 해결은 §6에서). 버전도 여러 세션이 동시에 bump해서, push 전 git fetch + rebase로 build.gradle·CHANGELOG 충돌을 매번 풀어야 했습니다. 병렬 개발은 공짜가 아니라, 락·격리·리베이스라는 대가를 치르고 산 속도였습니다.

자율의 경계를 사람과 맞추다 — 루프 엔지니어링의 진짜 어려움

Loop 엔지니어링에서 제일 까다로운 건 알고리즘이 아니라 "에이전트가 정확히 어디서 멈추고 사람에게 물어야 하는가" 입니다. 이 경계는 정적이지 않았습니다. 여러 번 질책받으며 조정됐습니다. 솔직하게 그 궤적을 옮기면:

① 이슈 사이에서 멈추지 마라. 처음엔 이슈 하나를 머지하고 "다음으로 진행하겠습니다"라고 보고한 뒤 턴을 끝냈습니다.

"왜 자꾸 멈추는거야"

자율 루프의 본질은 연속입니다. 머지 직후 곧바로 다음 이슈를 착수해야지, 보고+정지는 흐름을 끊는 노이즈였습니다.

② 당연한 결정을 떠넘기지 마라. 분석을 잔뜩 늘어놓고 "1번? 2번? 3번?" 선택지를 던지며 멈추는 버릇이 있었습니다.

"역지사지를 하라고" … "그 판단은 이제 니가 능동적으로 좀 하자" … "너가 자율개발모드 진행중인데 나 없다고 생각하고 진행해야지"

다음 스텝이 누가 봐도 당연하면(검증해야 함 / 추측 말고 1차 출처 확인 / 안 되면 고침) 묻지 말고 끝까지 하고 결과만 보고. 추측이 나오면 멈추는 게 아니라 즉시 웹 조사 + 실측으로 확정하는 게 기본 동작이 됐습니다.

③ 입력을 기다리며 정지하지 마라 — 폴링하라. PR을 push하고 사용자 입력을 기다리며 멈춰 있었습니다.

"왜 멈춰있어. 머지 대기 폴링 몇 분으로 설정한 거야"

이후로는 push 직후 명시적 폴링(예약 wakeup, 5분 미만) 을 걸거나 독립 이슈로 병렬 전환합니다. CI가 도는 동안 놀지 않습니다.

④ 리뷰 발견을 이슈로 도피하지 마라. 7관점 리뷰가 발견을 뱉으면 그걸 죄다 후속 이슈로 등록하고 넘겼습니다. 그 결과 한 세션에 이슈가 12건 생성 / 5~8건 종료 — 순증. 열린 이슈가 43개까지 불었습니다.

"github issue 가 줄어들지 않고 늘어나기만 하는데 대체 무슨일이야?" … "버그를 양성해놓고 이슈등록하고 나몰라라하는건 아냐"

하드 규칙이 섰습니다: 리뷰 발견은 기본 흡수. "지금 코드로 고칠 수 있나?"를 먼저 grep·Read로 확인하고, 고칠 수 있으면 이 PR에서 흡수. 특히 발견이 이슈의 수용 기준/제목 그 자체면 무조건 흡수(후속으로 빼면 제목값을 절반만 한 PR이 됨). 후속 이슈는 진짜 별개(새 트리거·큰 설계 변경)일 때만.

⑤ 그러나 자율에도 절대 경계가 있다. 한번은 열린 이슈를 자율로 골라 Prometheus 의존성 + 무인증 /prometheus 엔드포인트를 추가·머지했습니다.

"누구맘대로 넣은거야 이거 난생 처음보는데" … "당장 제거해"

의존성·인프라·외부 노출·시크릿은 이슈가 열려 있어도 사용자 동의 게이트입니다. 특히 7관점 리뷰가 보안/노출을 하드 블로커로 지목하면, disclosure 달고 우회 머지하는 건 금지. 이건 "당연한 다음 스텝"이 아니라 사용자가 통제하는 영역입니다.

이 다섯 가지가 제 루프의 human-in-the-loop 체크포인트를 정의합니다. 자율의 폭을 최대로 넓히되(①~④), 되돌리기 어려운 결정만 사람에게 올린다(⑤). "자율 ≠ 무제한"이고, 이 경계선을 찾는 것 자체가 Loop 엔지니어링이었습니다.

6. 운영이 루프를 시험한 순간들

"루프는 마법이 아니라 엔지니어링 산출물"이라는 말은, 현실이 끊임없이 루프를 시험한다는 뜻이기도 합니다. 3주간 부딪힌 대표적인 마찰들:

  • CI가 통째로 마비됐다 — 코드 문제가 아니었다. 어느 날 모든 CI job이 3~4초 만에 실패했습니다. 로그: "The job was not started because recent account payments have failed" — GitHub Actions 결제/지출한도 문제였습니다. required_status_checks라 CI 미통과 시 머지가 막히고, admin 머지조차 거부됐습니다(ruleset이 admin에게도 강제). 근본 해결로 워크플로를 내 맥북(self-hosted runner)으로 전환했는데, 그러려면 main의 워크플로를 바꿔야 하고 그건 또 ruleset이 막는 순환에 빠졌습니다. ruleset enforcement를 잠깐 disable → main push → 즉시 복원(보호 창 최소화)으로 탈출했습니다.

  • self-hosted runner가 OOM으로 죽었다. 내 로컬 gradle 데몬들과 메모리 경합. GRADLE_USER_HOME에 kotlin.compiler.execution.strategy=in-process + org.gradle.daemon=false를 박아 JVM 한 세트로 묶어 해결.

  • 옆 워크트리가 내 빌드를 죽였다. 두 워크트리가 gradle 데몬을 공유해서, 한쪽의 --stop이 다른 쪽 빌드를 중단시켰습니다. GRADLE_USER_HOME으로 데몬 registry만 격리(caches는 심볼릭 공유)해 회피.

  • 테스트가 간헐적으로 즉사했다. 임베디드 Mongo가 고정 포트(28017)를 써서, 동시에 도는 다른 워크트리 테스트와 충돌 → 전역 컨텍스트 기동 실패 → 전 테스트 abort. 랜덤 빈 포트로 전환해 해결.

  • kover 100%의 미로. 100% 커버리지를 진짜로 채우려니 직관에 반하는 함정이 수두룩했습니다 — 람다 경유로 호출되는 함수는 미커버로 잡히고, sealed when의 마지막 분기는 컴파일러가 만든 도달불가 else 때문에 영원히 99.9x%에 걸리고, Nothing 반환 헬퍼를 분기에서 호출하면 팬텀 fall-through edge가 남습니다(직접 throw로 바꾸면 즉시 통과). 이런 걸 하나씩 정리하며 커버리지 게이트를 유지했습니다.

  • 가장 무서운 건 "조용한 실패"였다. prod에서 임베딩 토글이 꺼져 있으면 뉴스·회상 벡터 검색이 전부 빈 결과를 냅니다 — 그런데 부팅은 정상입니다. 크래시가 아니라 성능만 조용히 죽어서, 실거래 중 관측으로 발견했습니다. 이후 "부팅 성공 = 정상"이라는 가정을 버렸습니다.

이 마찰들은 개발 방법론의 일부입니다. 루프를 굴린다는 건 이런 현실의 저항을 하나씩 게이트·격리·관측으로 길들이는 일이었습니다.

7. 이 방법이 만든 것 — 제품 miner의 내부

엔지니어링된 개발 루프가 3주간 실제로 무엇을 뽑아냈는지. (여기부터는 개발 방식이 아니라 산출물입니다.)

마일스톤 아크 — 수습에서 실거래까지

  • Phase 0 — M1 Foundation(토스 API·시세·캔들·Mongo Vector Search) → M2 Core Domain(Mandate·Decision·TrackRecord·RISK-GUARD) → M3.5 Three-Track(뉴스 dedupe·동적 universe) → M3.6 결재라인(교착 에스컬레이션).
  • Phase 1 (p1-m1~m7) — 데이터층(뉴스·매크로·DST 트리거) → 두뇌(실 임베딩·Vector Search 회상·EOD 회고·프롬프트 외부화) → 추론(tool-calling 루프·RuleEngine·structured output) → 집행(LIVE 실주문·투트랙 e2e) → 관측(LLM 비용·알람·회고 리포트) → 트레이더 지식(MACD·EMA·ATR·지식 카드 주입) → 인트라데이(모멘텀 tier: LLM 계획 + 결정론 FAST 집행).
  • 2026-06-30, prod가 mode=LIVE로 전환 — Phase 1(정직원) 진입, 소액 실거래 시작. (북극성 4단계는 §2 참조 — 종착지는 자가운영.)

트레이딩 시스템의 골격

판단하는 뇌(LLM·회상·페르소나)는 §4에서 다뤘으니, 여기선 그 뇌를 감싼 시스템만 봅니다. 중심은 agent harness — 깨어난 에이전트가 도구를 쥐고 "무엇을·언제·어떻게 매매할지"를 스스로 판단하는 구조입니다. 기획=판단 / 재무=집행으로 나뉘고, 다음에 언제 깨어날지는 Appointment로 스스로 예약하며(ByTime/ByCondition), 집행은 2-경로(느린 협상 / 급매 바이패스)로 갈리고, 마지막엔 결정론 RISK-GUARD(AI 미위임)가 막습니다. 이 구조는 초기의 '고정 절차' 설계를 7관점 3라운드로 전면 재설계해 얻었고, 그 과정에서 낡은 ADR 10개를 폐기했습니다 — 틀린 결정을 지우는 것도 개발입니다.

8. 돌아보며 — 가장 깊은 대칭

3주를 관통한 한 문장은 이것입니다:

나는 엔지니어링된 AI 개발 루프로, 엔지니어링된 AI 트레이딩 루프를 만들었다.

내 개발 방식과, 그 방식이 만든 제품이 정확히 같은 구조였습니다:

나의 개발 루프 (방법) miner의 트레이딩 루프 (제품)
깨우는 것 /goal 프롬프트 → 개발 에이전트 Appointment → 트레이딩 에이전트
판단자 LLM 개발자 (도구로 자율 구현) LLM 트레이더 (도구로 자율 판단)
재료 이슈·ADR·컨벤션 관측·회상·뉴스
진리원 docs/ 단일 진리원 영속 메모리(journal·brain)
검증(verify) kover 100% + 7관점 리뷰 재무 집행 심사
마지막 방어선 게이트(validate·CI·ruleset) RISK-GUARD (AI 미위임)
사람의 자리 게이트가 막거나 교착일 때만 교착 시 에스컬레이션

둘 다 "에이전트를 프로그래밍한다" 는 같은 사상 위에 있습니다 — 에이전트에게 재료와 도구를 충분히 주고, 자율적으로 반복하게 두되, 되돌리기 어려운 위험만 결정론 게이트로 막는다.

남은 교훈들

  1. 자율 규약은 휘발된다 → 게이트로 묶어라. 다짐은 잊히고, CI/hook에 박힌 규칙만 산다. 프롬프트는 얇게, 진리원은 문서로.
  2. 검증을 루프에 내장하라. "한 번에 맞히기"가 아니라 "해보고·확인하고·고치고·반복." 검증이 사람의 눈이 아니라 루프의 한 단계여야 자율이 성립한다.
  3. 자율의 경계는 찾아가는 것이다. 어디서 멈추고 물을지는 정적이지 않다 — 멈추지 말아야 할 곳(이슈 사이·당연한 결정)과 반드시 멈춰야 할 곳(인프라·시크릿·실 자금)을 질책받으며 조정했다.
  4. 검증을 사고의 대체재로 쓰지 마라. validate·리뷰는 최후 방어선이지 발견 수단이 아니다. 쓰기 전 파급을 grep하고, 커밋 전 diff를 자가 리뷰한다.
  5. 단일 시선은 사각을 만든다 → 다관점. 특히 실 자금이 걸린 도메인에선 7개 관점이 각자 다른 사고를 막는다.
  6. AI는 판단, 결정론은 방어선. 개발에선 게이트, 트레이딩에선 RISK-GUARD. 마지막 문은 절대 AI에 안 맡긴다.
  7. 루프는 마법이 아니라 산출물이다. CI 결제 차단, gradle 데몬 충돌, kover 함정, 조용한 실패 — 현실의 저항을 게이트·격리·관측으로 하나씩 길들이는 게 loop 엔지니어링의 실체다.

9. 무엇을 믿고, 무엇이 아직 미완인가

이 방식의 신뢰는 리뷰어가 아니라 실행된 동작에 걸려 있습니다. LLM이 짠 코드든 사람이 짠 코드든, 저는 e2e 테스트로 실제 사이클을 굴려 동작을 관측한 결과를 근거로 믿습니다. LLM이라서 특별히 더 의심하거나 더 믿지 않고, 숙련된 개발자가 한 것과 같은 잣대로 봅니다 — 사람이 짠 코드도 100% 안전하지는 않으니, 완벽을 증명하라는 게 아니라 테스트로 확인하면 됩니다. 그 아래를 결정론 게이트(커버리지·컴파일·CI)가 받치고, 7관점 리뷰는 사각을 줄이는 보조입니다. LLM이 LLM을 검증한다는 무한 거울의 착시는, 최종 판정을 말이 아니라 실행된 동작에 두는 순간 사라집니다.

그래도 게이트를 다 통과하며 확신에 차 틀리는 결함은 남습니다. 커버리지 100%가 정확성을 뜻하지는 않으니까요. 임베딩 토글이 꺼진 채 배포돼 벡터 검색이 조용히 빈 결과를 냈을 때, 부팅은 정상이라 게이트가 못 잡았고 실거래 관측으로야 발견했습니다. 이런 결함은 없앨 수 없고 가둘 뿐입니다 — 런타임 결함을 관측·알림으로 게이트화하고, 최종 안전망은 결정론(제품의 RISK-GUARD, 개발의 사람 교정)에 둡니다.

완료 신고는 그 자체로 믿지 않았습니다. 종료는 두 겹으로 확인합니다. 하나는 "머지·close"라는 외부 상태입니다 — 자기보고가 아니라 실제로 머지됐는가라는 결정론적 사실이죠. 진척 없이 게이트만 반복 실패하면 실패 신호로 보고, 예산을 다 쓰고도 CLEAN이 안 되면 멈춰 올립니다. 다른 하나는 재서술입니다. "다 됐습니다"라는 말이 나오면 한 작업을 다시 읊게 하되, 기존 요약을 읽어(read) 참조시키는 게 아니라 다시 써내게(write) 합니다. 능동적 재구성은 수동적 참조보다 맥락 이해도를 끌어올리고, 빠뜨린 부분·지어낸 부분을 그 자리에서 드러냅니다.

프롬프트를 얇게 유지할 수 있는 건 규칙이 프롬프트가 아니라 문서·게이트에 있기 때문입니다. 문서 규칙의 상당수가 validate·CI·Konsist로 강제돼 있어, 에이전트가 문서를 안 읽어도 대부분 게이트가 잡습니다. 문서는 "왜"를, 게이트는 "강제"를 맡습니다. 강제되지 않는 규칙이 자꾸 어겨진다면 그건 게이트로 박으라는 신호입니다. 없는 함수·필드를 지어내는 환각은 원문 직접 정독으로 막습니다 — 요약에 의존하지 않고, "필드가 없다" 같은 주장도 grep·Read로 확인한 뒤 행동하며, 지어낸 심볼은 컴파일과 자가 리뷰가 걸러냅니다. 규칙이 문서·게이트에 있는 덕에 모델 교체에도 덜 민감해서, 3주간 여러 모델 버전이 혼용됐어도(opus-4-7·4-8 등) 루프는 그대로 돌았습니다.

아직 손대지 못한 건 비용입니다. 이 단계의 초점이 금액과 무관하게 현재 가능한 퍼포먼스를 극대화하는 데 있었던 터라, 비용 스케일링은 깊이 고민하지 않았습니다. 캐시 read가 (맥락 × 턴)으로 커지는 건 사실이고 이슈를 작게 자른 덕에 부수적으로 비용에도 유리했지만, 그건 리뷰·머지 단위를 잘게 두려던 결과지 비용 최적화는 아니었습니다. 비용을 의도적으로 제어하는 일은 다음 단계의 과제로 남아 있습니다.

10. 3주의 실측, 그리고 해보려는 사람을 위한 메모

3주간의 세션 로그가 이 루프의 실제 모습을 숫자로 보여줍니다(메인 루프 20개 세션 기준이고, 서브에이전트 리뷰 토큰은 별도 로그라 합계엔 빠져 있습니다).

루프는 Claude Code(Opus)에서 /goal 명령 하나로 시작합니다. 3주간 22번의 킥오프가 82,959번의 assistant 턴을 낳았고(킥오프당 평균 약 3,800턴), 한 세션이 길게 돌아 최대 단일 세션은 17,135턴에 달했습니다. 컨텍스트 한도는 자동 압축으로 넘기고, 이슈가 쌓이는 속도가 빠르면 워크트리로 여러 세션을 병렬로 돌립니다. PR을 push한 뒤에는 폴링으로 CI를 기다리며 다음 이슈를 병렬 진행합니다.

/goal 프롬프트 자체는 원칙 한 줄이 골격입니다 — 규칙을 담지 않고 문서가 정의한 게이트를 집행하는 오케스트레이션일 뿐, 숫자·관점·기준이 필요하면 해당 문서를 읽어 따른다는 것. 본문은 여덟 블록의 절차입니다: 착수 전 복기, 정리(열린 PR 몰아 머지 + 이슈 크로스체크), 순서·브랜치, 이슈별 라이프사이클, 리뷰·흡수(관점 수는 프롬프트가 아니라 code-review.md에 위임), 자율·청취 경계, e2e 완료 검증, 루프, 개발 방식. 7관점 리뷰는 PR 생성 직후 서브에이전트 여럿을 병렬로 띄워 각자 diff와 ADR을 정독시키고 gh api ...pulls/{PR}/comments로 파일·라인에 inline suggestion을 달게 한 뒤, 메인이 코드를 직접 정독해 흡수와 후속을 판단하는 식으로 돌아갑니다. 서브에이전트는 관점 시뮬과 등록만 하고, 결정은 메인이 합니다.

자율성은 이렇게 나타납니다. 사람 프롬프트 사이 간격의 중앙값은 3.6분이지만 p90이 27분, 최대 12.7시간입니다. 30분 넘게 무개입으로 돈 구간이 246회, 한 시간 이상이 122회였고, assistant 턴과 사람 프롬프트의 비율은 약 31:1입니다. 대부분은 사람이 근처에 있었지만 수백 번은 장시간 방치로 돌았고, 그 구간의 품질을 게이트가 지켰습니다. 프롬프트 입력 시간대(KST)는 하루 24시간에 고루 퍼져 있고 저녁 21–22시와 심야 02시가 피크였습니다. 로그는 밤새 돌았고 사람은 저녁·심야에 몰아서 개입했으니, "혼자 3주"의 실체는 쉬지 않는 루프에 간헐적으로 조타를 넣는 일이었습니다.

비용 쪽 숫자도 흥미롭습니다. 메인 루프가 태운 토큰은 약 42.1B(420억)인데, 그중 99%(41.8B)가 캐시 read이고 실제 생성된 출력 토큰은 92.6M(0.2%)뿐입니다. 루프가 매 턴 커져가는 맥락을 다시 읽으므로 캐시 read가 폭증하지만, 캐시 read는 신규 입력의 1/10 단가라 토큰 총량 대비 훨씬 쌉니다. 리스트가로 환산하면 수만 달러 규모지만 실제 지출은 구독제 정액이라 그와는 무관합니다.

시작하려는 사람에게 가장 중요한 건 선행 조건입니다. 이게 없으면 자율이 성립하지 않습니다 — 테스트·커버리지·lint를 한 명령으로 묶은 검증 게이트(pre-push hook 포함), 우회 불가능한 main ruleset, 문서 단일 진리원, 그리고 "의도"로 쓴 이슈. 다만 처음부터 전부 갖출 필요는 없습니다. 게이트 하나(테스트 통과)만 있어도 "이슈→코드→테스트→PR→머지" 루프는 돌고, 게이트는 발견할 때마다 더하면 됩니다. 이 방식은 솔로·그린필드, 타입이 강한 언어, 명확한 도메인, 무엇보다 검증 가능한 종료조건이 있을 때 유리하고, 게이트를 세우기 힘든 레거시나 검증 불가능한 작업, 락이 필요한 팀 병렬에서는 어렵습니다. 관건은 언어나 규모가 아니라 "테스트로 못박을 수 있는가"입니다.
